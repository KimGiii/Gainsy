## 아쉬운 것 (Bad)

- **문서 제거 자동화가 한 번 더 비용을 만들었다**: prod 문서 제거와 복원 흐름이 반복되면서 문서 보존 전략을 다시 손봐야 했다. 배포 브랜치와 지식 저장소를 같은 방식으로 다루기 어렵다는 점이 드러났다.
- **디자인 수정 범위가 컸다**: 토큰화와 다크모드 개선은 필요했지만, 화면 전반에 걸쳐 많은 파일을 건드렸다. 회귀 점검 체크리스트와 lint가 생긴 만큼 이후에는 작은 단위로 유지해야 한다.
- **권장량 라벨 UX가 한 번에 정리되지 않았다**: 홈과 식단 화면에서 `권장 X` 표시가 중복되어 후속 커밋으로 제거했다. 숫자 정보는 많아질수록 위계가 중요하다.
- **캐시 정합성 문제가 여러 도메인에서 반복됨**: 신체 측정, 목표 생성·포기, 영양 권장량이 서로 연결되면서 캐시 evict 규칙을 더 명확한 도메인 이벤트로 정리할 필요가 생겼다.

## 배운 것 (Learned)

- **디자인 시스템은 컴포넌트보다 검출 장치가 중요하다**: 토큰을 만들어도 새 인라인 값이 계속 들어오면 금방 흐트러진다. `design-lint`를 fail-on-new로 올린 것이 실제 전환점이었다.
- **푸시 라우팅은 앱 생명주기와 경쟁한다**: 알림 탭 시점, 앱 초기화, 라우터 준비가 엇갈리면 목적 화면으로 가지 못한다. 전용 `PushRouter`로 의도를 보관하고 준비된 뒤 소비하는 구조가 필요하다.
- **권장 영양량은 목표 변경과 함께 움직여야 한다**: 회원가입 시 한 번 계산하는 값이 아니라 목표 생성·포기, 신체 정보 변경에 따라 다시 계산되어야 한다.
- **캐시 장애는 보조 기능 실패로 취급해야 한다**: Redis가 흔들려도 핵심 API가 같이 죽으면 사용자 입장에서는 전체 서비스 장애다. 캐시는 성능 계층이지 단일 장애점이 되어서는 안 된다.
