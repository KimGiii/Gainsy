- **출시 심사 대응과 품질 개선 병행**: App Store 거절 대응을 하면서 UI 테스트와 업로드 실패 UX까지 보강했다. 심사 통과만 목표로 삼지 않고 실제 사용 실패 지점까지 같이 막았다.
- **도메인 정합성 발견**: 목표, 신체 측정, 사용자 프로필이 서로 다른 진실 공급원이 되는 문제를 잡았다. 이후 권장 영양량 자동 계산으로 이어질 기반이 생겼다.
- **문서화 밀도**: 코드 리뷰 결과, 재심사 대응, 영양소 표준 변경을 문서에 남겨 후속 작업자가 맥락을 빠르게 복원할 수 있게 했다.

## 아쉬운 것 (Bad)

- **prod 문서 제거 전략 시행착오**: GitHub 웹 머지가 사용자 정의 merge driver를 무시한다는 점을 뒤늦게 확인했다. prod 전용 문서 제거 자동화는 Git 속성보다 워크플로우나 배포 산출물 분리로 접근했어야 했다.
- **보안 리뷰가 출시 직전에 몰림**: 인증, CORS, 토큰 만료, 페이징 상한 같은 항목은 초기 백엔드 뼈대 단계에서 기준으로 고정했으면 비용이 작았을 것이다.
- **영양소 표준 변경의 파급이 큼**: 백엔드 DTO, DB, iOS 모델, UI 표시까지 한 번에 맞춰야 해서 작은 스키마 변경처럼 보이지만 실제 영향 범위가 넓었다.
- **테스트 통과 기준의 잔여 구멍**: 백엔드 리뷰 당시 대부분 테스트는 통과했지만 DB 의존 컨텍스트 로드 테스트 등 환경 의존 케이스는 여전히 별도 정리가 필요했다.

## 배운 것 (Learned)

- **컨트롤러 인증은 주입으로 통일해야 한다**: JWT 문자열을 각 컨트롤러에서 직접 파싱하면 검증 경로가 분산되고 우회 위험이 생긴다. 인증 필터 이후에는 `SecurityContext` 기반 주입으로 일관성을 유지하는 편이 안전하다.
- **조회 API는 쓰기를 하지 않는 것이 운영상 유리하다**: GET에서 누락 체크포인트를 보강하면 편해 보이지만, 동시성 중복 INSERT와 예측 불가능한 DB 쓰기가 생긴다. 보강 작업은 생성 시점이나 스케줄러로 분리하는 편이 낫다.
- **GitHub 웹 머지는 로컬 Git 설정을 믿으면 안 된다**: `.gitattributes`와 사용자 정의 merge driver는 로컬/CLI 맥락에서는 의미가 있지만 GitHub 웹 머지 정책에는 그대로 적용되지 않을 수 있다.
- **심사 대응 문서는 작업 로그가 아니라 판단 근거여야 한다**: 어떤 거절 사유에 어떤 변경으로 대응했는지 남겨야 재심사 노트와 후속 릴리즈 준비가 빨라진다.
